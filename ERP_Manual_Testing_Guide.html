Pre-Flight Setup

Before testing, ensure your environment is synchronized and the API is healthy.

Database & Migrations
Run this in your terminal to ensure the schema is up-to-date: 
dotnet ef database update --project ERP_System.Infrastructure --startup-project ERP_System.API
Launch the System
Start the API and wait for the Swagger UI to open: 
dotnet run --project ERP_System.API
Access URL: https://localhost:5001/swagger

The Transactional Test (The "Big One")

This tests CQRS, MediatR, Transactions, Repositories, and Stock Deduction in one flow.

1. Create Order
{
  "customerId": 1,
  "notes": "Testing workflow",
  "items": [{ "productId": 1, "quantity": 5 }]
}
2. Verify Side Effects
  • Stock Level: Check GET /api/stock. Quantity should be 45 (Init 50 - Order 5).
  • Response Detail: Check GET /api/orders/{id}. Ensure Customer and Product names map correctly via AutoMapper.
  • Failure Case: Try to create the same order with quantity: 100. Verify you get an InsufficientStockException (HTTP 422).
4

How Everything is Connected

The system is built on Clean Architecture. Here is the lifecycle of a request:

1. API Layer (Entry)

Controllers receive HTTP requests. They use MediatR to send Commands/Queries to the Application layer. Zero business logic here.

2. Application Layer (Brain)

Handlers coordinate the work. They use FluentValidation for rules, AutoMapper for DTOs, and Repositories for data.

3. Domain Layer (Core)

Rich Entities (Product, Order) enforce core business rules. This is the heart of your ERP system.

4. Infrastructure (Storage)

EF Core handles the heavy writes. Dapper provides lightning-fast reads. Unit of Work keeps it all atomic.

The Master Testing Roadmap

Run these in the exact order shown below to build a full business scenario.

STEP 01 Category: Create a grouping (e.g., "Electronics")
STEP 02 Supplier: Register where you buy products from
STEP 03 Product: Create a product linked to Category (ID from Step 1)
STEP 04 Warehouse: Define a storage location (e.g., "Main Warehouse")
STEP 05 Inventory: Add stock for Product (ID from Step 3) in Warehouse
STEP 06 Customer: Register a buyer
THE GOAL: STEP 07 Order: Create a sale. It checks Customer + Product + Subtracts Stock.
STEP 08 Employee: Register the user handling the transactions

Detailed Endpoint Workflow Library

1. CategoriesController — [Configuration Module]

Goal: Setup the high-level grouping. Everything in the system starts here.

POST /api/categories
{
  "categoryName": "Laptop",
  "description": "Premium compute devices",
  "isActive": true
}
➔ Run GET /api/categories/all to get the categoryId.
2. SuppliersController — [Supply Chain]

Goal: Register the vendors who provide you with goods.

POST /api/suppliers
{
  "supplierName": "TechCorp Global",
  "contactPerson": "Alice Smith",
  "email": "supply@techcorp.com",
  "phoneNumber": "+1-800-SUPPLY",
  "address": "456 Silicon Valley",
  "isActive": true
}
3. ProductsController — [Catalog Module]

Goal: Initialize your item master. Requires categoryId from Step 1.

POST /api/products/create
{
  "productName": "MacBook Pro M3",
  "sku": "MBP-M3-01",
  "price": 1999.99,
  "costPrice": 1500.00,
  "categoryId": 1,
  "description": "Late 2024 model"
}
➔ Run GET /api/products/all-products to get the productId.
4. WarehousesController — [Logistics]

Goal: Create a physical location where stock is held.

POST /api/warehouses/add-warehouse
{
  "name": "Dubai Hub",
  "location": "Jebel Ali Freezone",
  "isActive": true
}
5. InventoriesController — [Operations]

Goal: Add actual quantities. Requires productId (Step 3) and warehouseId (Step 4).

POST /api/inventories/add-stock
{
  "productId": 1,
  "warehouseId": 1,
  "quantity": 50
}

Low Stock Monitoring:

GET /api/inventories/low-stock
➔ Note: Returns any product where total stock across all warehouses is below 10.
6. CustomersController — [CRM Module]

Goal: Setup the buyer profile. Necessary for order attribution.

POST /api/customers
{
  "customerName": "John Doe",
  "email": "john.doe@email.com",
  "phoneNumber": "555-0101",
  "address": "123 Main St",
  "isActive": true
}
➔ Note: GET /api/customers will show the ID needed for the order.
7. OrdersController — [Transaction Module]

Goal: The final transaction. Combines customerId (Step 6) and productId (Step 3).

POST /api/orders
{
  "customerId": 1,
  "notes": "Premium Packaging please",
  "items": [
    { "productId": 1, "quantity": 1 }
  ]
}

Order Management & Reporting:

PATCH /api/orders/{id}/status
Body: 2 (Integer for Shipped, 3 for Delivered, etc.)

GET /api/orders/reports/sales?from=2024-01-01&to=2024-12-31

Behind the Scenes: Creating an order validates Customer/Product status and subtracts stock. The Sales Report joins Orders and Items to calculate daily revenue.

8. EmployeesController — [HR Module]

Goal: Register the workforce. Can be extended for "Order Assigned To" logic.

POST /api/employees
{
  "firstName": "Robert",
  "lastName": "Manager",
  "email": "robert@erp.com",
  "phoneNumber": "555-9090",
  "position": "Sales Manager",
  "department": "Logistics"
}

Developer Troubleshooting Hub

Error ProfileThe "Hidden" ReasonSolution
Conflicted with FK Constraint "FK_Products_Categories" EF Core is trying to re-update the Category while updating the Product. Set Entry(cat).State = Unchanged in Repo.
No route matches supplied values Action name has "Async" suffix stripped by ASP.NET Core by default. Use CreatedAtRoute with named route instead of CreatedAtAction.
Product is not active (even in DB) Dapper Column Ordering bug (IsActive mapped to Category). Ensure IsActive comes BEFORE CategoryId in SELECT query.

Business Logic: Roles & Responsibility

Why are Employees in the system?

In a real ERP, **Employees** represent the "Who". While the system currently stores their profiles, they are meant for **Accountability and Assignment**:

  • Order Fulfillment: Assigning an employee to "Pick" an order.
  • Audit Logs: Tracking which employee created a product or updated stock.
  • Departmental Security: Restricting access based on their position (e.g., only "Sales Manager" can approve large discounts).
Who updates the Order Status?

The Logistics/Warehouse Team is responsible for calling the PATCH /api/orders/{id}/status endpoint:

  • Warehouse Staff: Marks it as 2 (Processing) when they start picking items.
  • Dispatch Team: Marks it as 3 (Shipped) when the courier picks it up.
  • Delivery System: Automatically marks it as 4 (Delivered) via a callback when the customer signs for it.
