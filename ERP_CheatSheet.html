📐 The Golden Rule

Dependencies always point inward. Domain knows nothing about Infrastructure. Application knows nothing about Controllers.

Why? Swap EF Core for Dapper — only the Infrastructure layer changes. The rest of the app is untouched.

CQRS Pattern + MediatR

Pattern

✍️ Command (Write Side)

Commands mutate state. Named as VerbNounCommand. Returns the created/modified resource.

public record CreateProductCommand(
    string Name,
    decimal Price,
    int CategoryId
) : IRequest<ApiResponse<ProductResponseDto>>;

public class CreateProductCommandHandler
    : IRequestHandler<CreateProductCommand, ...>
{
    public async Task<...> Handle(...)
    {  /* business logic + repository call */  }
}
📁 Application/Features/Products/Commands/CreateProduct/

🔍 Query (Read Side)

Queries only read data. Never change state. Named as GetNounQuery or SearchNounQuery.

public record GetProductByIdQuery(int Id)
    : IRequest<ApiResponse<ProductResponseDto>>;

public class GetProductByIdQueryHandler
    : IRequestHandler<GetProductByIdQuery, ...>
{
    public async Task<...> Handle(...)
    {  /* just reads, never writes */  }
}
📁 Application/Features/Products/Queries/GetProductById/

Repository Pattern + Unit of Work

Pattern

📂 Repository Pattern

Each entity has its own repository interface in Domain and concrete implementation in Infrastructure. The handler only talks to the interface — never directly to EF Core.

// Domain — pure interface
public interface IProductRepository
{
    Task<Product> GetByIdAsync(int id);
    Task AddAsync(Product product);
    ...
}

// Infrastructure — actual EF Core logic
public class ProductRepository : IProductRepository
{
    private readonly AppDbContext _ctx;
    ...
}
📁 Domain/Interfaces/IProductRepository.cs📁 Infrastructure/Persistence/Repositories/ProductRepository.cs

🔗 Unit of Work (UoW)

Wraps multiple repository operations into a single DB transaction. Ensures all-or-nothing saves.

public interface IUnitOfWork
{
    Task BeginTransactionAsync();
    Task<int> SaveChangesAsync();
    Task CommitAsync();
    Task RollbackAsync();
}
// Usage in Handler:
await _uow.BeginTransactionAsync();
// ... do work ...
await _uow.SaveChangesAsync();
await _uow.CommitAsync();
📁 Domain/Interfaces/IUnitOfWork.cs📁 Infrastructure/Persistence/UnitOfWork.cs

💡 Why separate Interface (Domain) from Implementation (Infrastructure)?

ConcernBenefit
TestabilityMock IProductRepository in unit tests without needing a real DB
FlexibilitySwap EF Core for Dapper or another ORM by only changing Infrastructure
Dependency RuleApplication layer only depends on Domain — never on EF Core directly

Dapper (Micro-ORM) + ASP.NET Core Web API

Database + API

⚡ Dapper — When & Why

Used alongside EF Core for performance-critical or complex raw SQL queries. EF Core handles writes; Dapper handles complex reads.

public class DapperContext
{
    private readonly string _conn;
    public IDbConnection CreateConnection()
        => new SqlConnection(_conn);
}

// Usage:
var sql = "SELECT * FROM Products WHERE...";
var result = await conn.QueryAsync<Product>(sql);
📁 Infrastructure/Persistence/Context/DapperContext.cs

🌐 ASP.NET Core Routing

Controller-based routing with [Route] and [Http*] attributes.

[ApiController]
[Route("api/[controller]")]
public class ProductsController : ControllerBase
{
    [HttpGet("{id}", Name = "GetProductById")]
    public async Task<IActionResult> GetAsync(int id)

    [HttpPost("create")]
    public async Task<IActionResult> CreateAsync(...)

    [HttpPut("{id}")]
    [HttpDelete("{id}")]
}

📍 CreatedAtRoute — The Fix

Problem: CreatedAtAction(nameof(GetAsync), ...) fails because ASP.NET Core strips the Async suffix from action names in the route table by default (SuppressAsyncSuffixInActionNames = true).

// ❌ WRONG — "GetAsync" not in route table
return CreatedAtAction(nameof(GetAsync), ...);

// ✅ CORRECT — use named route
[HttpGet("{id}", Name = "GetProductById")]
...
return CreatedAtRoute(
    "GetProductById",
    new { id = result.Data!.Id },
    result);
Key Insight: CreatedAtRoute resolves by route name, not action name — immune to the Async suffix stripping issue.

MediatR Behaviors + AutoMapper + Validation

Cross-Cutting Concerns

🔗 Pipeline Behaviors (Middleware for Handlers)

Behaviors wrap every request/handler pair — like middleware but at the MediatR level. Run in registration order.

public class ValidationBehavior<TReq, TRes>
    : IPipelineBehavior<TReq, TRes>
{
    public async Task<TRes> Handle(
        TReq req, RequestHandlerDelegate<TRes> next, ...)
    {
        // validate BEFORE calling next()
        await next(); // calls actual handler
    }
}
📁 Application/Common/Behaviors/ValidationBehavior.cs📁 Application/Common/Behaviors/LoggingBehaviors.cs

🗺️ AutoMapper

Maps between domain Entities and DTOs (Data Transfer Objects). No manual property copying.

public class ProductMappingProfiles
    : Profile
{
    public ProductMappingProfiles()
    {
        CreateMap<Product, ProductResponseDto>();
        CreateMap<CreateProductCommand, Product>();
    }

// Usage in handler:
var dto = _mapper.Map<ProductResponseDto>(product);

✅ FluentValidation

Validators defined as classes, run automatically via ValidationBehavior before the handler executes.

public class CreateProductCommandValidator
    : AbstractValidator<CreateProductCommand>
{
    public CreateProductCommandValidator()
    {
        RuleFor(x => x.Name)
            .NotEmpty()
            .MaximumLength(100);
        RuleFor(x => x.Price)
            .GreaterThan(0);
    }
}

📦 ApiResponse<T> — Wrapper

All endpoints return a consistent ApiResponse<T> with Success, Message, and Data fields.

public class ApiResponse<T>
{
    public bool Success { get; set; }
    public string Message { get; set; }
    public T? Data { get; set; }

    public static ApiResponse<T> Ok(T data)
        => new() { Success = true, Data = data };
    public static ApiResponse<T> Fail(string msg)
        => new() { Success = false, Message = msg };
}

Infrastructure

🔌 Dependency Injection (DI)

Each layer exposes an extension method to register its own services. Program.cs calls all of them cleanly.

// Program.cs — clean startup
builder.Services.AddInfrastructureServices(config);
builder.Services.AddApplication();
builder.Services.AddControllers();
builder.Services.AddMemoryCache();

// ApplicationServices.cs registers:
// MediatR, FluentValidation, Behaviors

// InfrastructureServices.cs registers:
// DbContext, Repositories, UnitOfWork
📁 Application/DependencyInjection/ApplicationServices.cs📁 Infrastructure/DependencyInjection/InfrastructureServices.cs

📖 Swagger / OpenAPI

Auto-generates interactive API documentation from controller attributes and XML comments.

// Registration
builder.Services.AddSwaggerGen(c => {
    c.SwaggerDoc("v1", new() {
        Title = "ERP System API",
        Version = "v1"
    });
});
// Middleware
app.UseSwagger();
app.UseSwaggerUI(c =>
    c.RoutePrefix = string.Empty); // serves at /
📁 API/Program.cs — served at root URL

🛡️ Exception Middleware

Global error handler. Catches all unhandled exceptions and returns a structured JSON error response instead of a 500 crash page.

// Program.cs — registered FIRST in pipeline
app.UseMiddleware<ExceptionMiddleware>();

// Inside ExceptionMiddleware:
try { await _next(context); }
catch (Exception ex) {
    // write structured error JSON
}
📁 API/Presentation/ — ExceptionMiddleware.cs

🌍 CORS

Cross-Origin Resource Sharing — allows the API to be called from frontend apps on different domains.

// Policy registration
builder.Services.AddCors(o =>
    o.AddPolicy("AllowAll", p =>
        p.AllowAnyOrigin()
         .AllowAnyMethod()
         .AllowAnyHeader()));

// Applied in pipeline
app.UseCors("AllowAll");
📁 API/Program.cs

What You Can Add & Learn Next

Roadmap

🔑 JWT Authentication & Authorization

What: Secure API endpoints with JSON Web Tokens. Users log in, get a signed token, and pass it in every request.

// Program.cs
builder.Services.AddAuthentication(JwtBearerDefaults.AuthScheme)
  .AddJwtBearer(o => o.TokenValidationParameters = ...);

// Controller
[Authorize]           // 🔒 any logged-in user
[Authorize(Roles = "Admin")]  // 🔒 admin only
📁 Add: Infrastructure/Auth/JwtTokenService.cs 📦 Package: Microsoft.AspNetCore.Authentication.JwtBearer

🟥 Redis — Distributed Caching

What: Cache frequently-read data (product lists, categories) in an in-memory store. Orders of magnitude faster than hitting the DB every time.

// Register
builder.Services.AddStackExchangeRedisCache(o =>
    o.Configuration = "localhost:6379");

// Use in Handler
var cached = await _cache.GetStringAsync("products");
if (cached != null) return JsonConvert.Deserialize(cached);
📁 Add: Infrastructure/Caching/RedisCacheService.cs 📦 Package: Microsoft.Extensions.Caching.StackExchangeRedis

🚦 Rate Limiting

What: Protect your API from abuse by limiting how many requests a client can make per time window. Built into ASP.NET Core 7+.

// Program.cs
builder.Services.AddRateLimiter(o =>
    o.AddFixedWindowLimiter("fixed", opt => {
        opt.Window = TimeSpan.FromSeconds(10);
        opt.PermitLimit = 20;
    }));
app.UseRateLimiter();

[EnableRateLimiting("fixed")]  // on controller
📁 Add config in: API/Program.cs 📦 Package: Built-in (System.Threading.RateLimiting)

⏰ Background Jobs — Hangfire

What: Run scheduled or queued tasks outside the HTTP request cycle. Great for: sending order confirmation emails, generating daily sales reports, stock alerts.

// Register
builder.Services.AddHangfire(c =>
    c.UseSqlServerStorage(connStr));

// Enqueue a job
BackgroundJob.Enqueue(
    () => emailService.SendOrderConfirmation(orderId));

// Recurring job
RecurringJob.AddOrUpdate(
    "daily-report", () => ..., Cron.Daily);
📁 Add: Infrastructure/Jobs/ 📦 Package: Hangfire.AspNetCore + Hangfire.SqlServer

📡 SignalR — Real-Time Notifications

What: Push live updates from server to client over WebSockets. Ideal for: low-stock alerts dashboard, live order status, inventory level changes.

// Hub
public class StockHub : Hub
{
    public async Task SendLowStockAlert(string msg)
        => await Clients.All.SendAsync("LowStock", msg);
}
// Register + map
builder.Services.AddSignalR();
app.MapHub<StockHub>("/hubs/stock");
📁 Add: API/Hubs/StockHub.cs 📦 Package: Microsoft.AspNetCore.SignalR (built-in)

📄 Pagination + Sorting + Filtering

What: Return large datasets in pages instead of all at once. Add a PagedResult<T> wrapper and accept pageNumber/pageSize in queries.

public record GetAllProductsQuery(
    int Page = 1,
    int PageSize = 10,
    string? SortBy = null
) : IRequest<...>;

// EF Core
query.Skip((page - 1) * pageSize).Take(pageSize);
📁 Add: Application/Common/PagedResult.cs 📦 Package: X.PagedList or built-in LINQ

🧪 Unit Testing + Integration Testing

What: Test Handlers in isolation using mocked repositories. Clean Architecture makes this easy — handlers only depend on interfaces.

// xUnit + Moq
var repoMock = new Mock<IProductRepository>();
repoMock.Setup(r => r.GetByIdAsync(1))
        .ReturnsAsync(fakeProduct);

var handler = new GetProductByIdQueryHandler(
    repoMock.Object, mapperMock.Object);
var result = await handler.Handle(query, ct);
📁 Add: ERP_System.Tests/ project 📦 Package: xUnit + Moq + FluentAssertions

💓 Health Checks + Serilog Logging

What: Health Checks expose a /health endpoint for monitoring (DB alive? Redis alive?). Serilog replaces the default logger with structured, queryable logs.

// Health Checks
builder.Services.AddHealthChecks()
  .AddSqlServer(connStr)
  .AddRedis("localhost:6379");
app.MapHealthChecks("/health");

// Serilog
Log.Logger = new LoggerConfiguration()
  .WriteTo.Console()
  .WriteTo.File("logs/erp.log")
  .CreateLogger();
